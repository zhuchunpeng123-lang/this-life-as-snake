## 三、底层引擎改动流程（03_core.js / 04_collision.js）

2. AI 动手前必须显式告知"将改动 core/collision，影响 X/Y/Z，请确认"；
3. 只改必要行，不顺手重构；
4. 改完逐条回归所有依赖系统行为；
5. 优先用 Bus 事件 / Registry 扩展点，避免直接改底层。

## 四、规划模板（每次提优化目标先用此模板产计划，存 docs/plans/）

## 五、验收流程（改完后逐条核对）

- 逐条对照验收标准标 ✅/❌，写清复现步骤；
- 确认没改到计划外的文件；
- 确认 core.js / collision.js 未被意外改动；
- 全过才提示 git commit。

## 六、项目铁律（默认倾向，可带理由偏离）

- 数值只走 CONFIG（config.js 注释："GDD 仅给意图，禁直填裸数字"），业务代码不写裸数字。
- 系统间只通过 Bus('系统:动作') 通信，不跨层直调。
- 改动前先想清楚，歧义列出来让我选，不替我猜。
- **伤害一致性**：所有伤害经 `Core.Formula.damage(base, GS.segments, crit)`；技能层用 `hurt()` 包装，下游不自己算伤害。
- **DOT 语义**：持续伤害（`isDot=true`）只结算血量，按 `DOT_TEXT_MIN` 聚合飘字，不共用离散命中的物理反应（不击退、不刷硬直/闪白）。

## 七、数值真理源 §9 流程

- config.js 的数值来自《数值真理源 v0.3》§9。改任何**影响强度/平衡**的数值（伤害/血量/速度/射程/触发频率/掉率/倍率）：
  先更新真理源 §9 → 同步 config.js → 代码引用 CONFIG → 登记 §9 Changelog（顺序不可反）。
- 真理源尚未量化的**纯表现/节奏**值：就近用 `🟡 // TODO` + 2 候选占位，并在文件顶部 🟡 区 / 修改日志标注"待回写 §9 的债务"。
- 判不准是否影响强度 → 按 gameplay 处理（宁可登记进 §9）。
- 纯数值调参（不改结构与逻辑）走专用轻量通道，见 **第十节**；本节只规定真理源回写顺序，不重复 §八 计划闸。

## 八、改动前必先出计划（🔒 硬墙，不可偏离）

任何对代码的改动（含性能优化、玩法调整、去技术债），动手前必须：

1. 先产出《计划》文本（用第四节模板）：目标 / 涉及文件及理由 / 具体改动点 / 不动的底层 / 验收标准。
2. 等我确认计划后，才切换到可执行模式落地代码。
3. 落地后产出《测试清单》（复现步骤 + 预期结果），供我在浏览器手动复验。
4. 计划外文件一律不改；确需扩大范围时，先补计划让我确认。

此规则优先级高于"泛化偏离"——泛化能力用于**如何更好地实现计划**，不用于跳过"先确认"这一步。

纯数值调参（仅改 config.js 数值 / 仅用 ~ 调参器试探，不动结构逻辑）走 **第十节** 轻量通道，不重复本硬墙；仅当调参牵动公式/结构/伤害管线时，升级为本节出完整 §四 计划。

## 九、多窗口协作约定（推荐模式，非强制）

复杂改动建议拆为两个窗口，分离"作者"与"审查者"：

| 窗口 | 角色 | 职责 |
|------|------|------|
| **窗口 A**（规划+实现） | 作者 | Ask 出计划 → 你确认 → Craft 落地代码 → 输出 `docs/plans/*.md` + 测试清单 |
| **窗口 B**（校对） | 审查者 | 读 A 产出的计划 md + 测试清单 md + 被改代码文件 → 静态核对 → 输出差异报告（只报告、不擅自改代码） |

**交接协议**：
- A → B 只交文件，不靠聊天粘贴；被改文件路径写入计划 md，B 按路径读取。
- B 只做静态审查（代码 vs 计划、清单 vs 验收标准、裸数字/规则违反），不做浏览器实测。
- 浏览器手动复验由你完成。

单窗口模式同样可用（上下文热、步骤连贯），B 的存在是为了消除"实现者锚定偏见"和隔离上下文膨胀。

## 十、数值调参 / 平衡专用流程（激活量化 + 泛化的轻量通道）

本流程是 §七 真理源流程在"纯数值调参"场景下的**专用轻量通道**，与 §八 硬墙互补：它分流"改值"与"改结构"，但**绝不降级 AI**。核心立场——AI 在调参中是主动的量化与平衡伙伴，不是抄写员。

### 0. 触发判定（与 §八 的边界）
- 仅改 config.js 数值 / 仅用 ~ 调参器运行时试探 → 走本流程（**不触发** §八 厚计划）。
- 数值**结构变化**或牵动公式/逻辑（如改 `Core.Formula`、加字段、改伤害管线） → 升级为 §八 代码改动，出完整 §四 计划。
- 判不准 → 宁按 §八（更严）处理。

### 1. AI 必须主动量化（🔒 禁止降智）
- 用户给"手感 / 意图"描述（如"敌人太肉""前期太难"）时，AI **必须推导**出具体候选数值，而不是等用户报数。推导要可追溯：基准值 → 目标 delta → 公式依据。
- 每个调参提案至少给 **≥2 个候选数值**，各附权衡（手感曲线、难度坡度、对下游系统的影响）。
- 必须用 `Core.Formula` / Bus 事件链路做**波及分析**：这次改值会影响哪些系统、是否被 §六 伤害一致性约束覆盖。
- 真理源 §9 未量化的值：按 §七 用 `🟡 // TODO` + 2 候选占位，标注待回写债务。

### 2. 反降智护栏（显式禁令）
- ❌ 禁止"用户说 a 改成 b 就直接抄写"：即便接受用户给定数值，也必须附**推导 / 校验**，确认不破坏 §9 一致性与 §六 铁律（业务代码不写裸数字、伤害走 `Formula.damage`）。
- ❌ 禁止跳过量化：不允许只回"已调整"而不给具体数字 + 可复核的理由。
- ❌ 禁止盲从也禁止静默否决：AI 若认为用户数值不合理，**必须提出更优候选并说明理由**（激活泛化），明示分歧点让我选，不替我猜也不闷头照做。
- ✅ 鼓励：基于运行时数据 / 手感做沙盘推演，主动指出"改 X 会顺带让 Y 失衡"的连锁效应——这是泛化能力的用武之地。

### 3. 轻量提案模板（替代 §四 厚计划，但不可无）
《调参提案》=
1. 目标一句话（手感意图，非裸数字）
2. 候选数值与推导（≥2，含基准 / delta / 公式）
3. 波及分析（经 Bus / Formula 受影响的系统）
4. §9 真理源回写项（新增 / 修订哪条）
5. 验收标准（正常手感 + 至少 1 边界 / 反向场景）

### 4. 落地顺序（呼应 §七，不可反）
~ 调参器运行时试探 → 产《调参提案》让我确认 → 先回写 §9 真理源 → 同步 config.js → 登记 §9 Changelog → 给测试清单（复现步骤 + 预期手感）。

### 5. 多窗口用法（呼应 §九）
- 可新开"平衡窗口"专做本流程（上下文干净、防数字淹没主窗口），**但共用同一套 §十 规则，不得放宽**。
- 平衡窗口是"走轻量通道的窗口"，不是"豁免窗口"；需要升级 §八 时，照常出完整计划。
