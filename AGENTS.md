## 六、项目铁律（默认倾向，可带理由偏离）

- 数值只走 CONFIG（注释："GDD 仅给意图，禁直填裸数字"），业务代码不写裸数字。
- 系统间只通过 `Bus('系统:动作')` 通信，不跨层直调。
- 改动前先想清楚，歧义列出来让我选，不替我猜。
- **伤害一致性**：所有伤害经 `Core.Formula.damage(base, GS.segments, crit)`；技能层用 `hurt()` 包装，下游不自己算伤害。
- **DOT 语义**：`isDot=true` 只结算血量、按 `DOT_TEXT_MIN` 聚合飘字，不击退/不硬直/不闪白；分源 `dotMap` 独立聚合并传 `src`。细节与踩坑见 RETRO §9。

## 七、数值真理源 §9 流程

- 改任何**影响强度/平衡**的数值（伤害/血量/速度/射程/触发频率/掉率/倍率）：先更新真理源 §9 → 同步 config.js → 代码引用 CONFIG → 登记 §9 Changelog（顺序不可反）。

- 判不准是否影响强度 → 按 gameplay 处理（宁可登记进 §9）。
- 纯数值调参（不改结构逻辑）走 §十 轻量通道；本节只规定回写顺序，不重复 §八 计划闸。

## 八、改动前必先出计划（🔒 硬墙，不可偏离）

任何对代码的改动（含性能优化、玩法调整、去技术债），动手前必须：

1. 先产出《计划》文本（用 §四 模板）：目标 / 涉及文件及理由 / 具体改动点 / 不动的底层 / 验收标准。
2. 等我确认计划后，才切换到可执行模式落地代码。
3. 落地后产出《测试清单》（复现步骤 + 预期结果），供我在浏览器手动复验。
4. 计划外文件一律不改；确需扩大范围时，先补计划让我确认。

此规则优先级高于"泛化偏离"——泛化能力用于**如何更好地实现计划**，不用于跳过"先确认"这一步。

纯数值调参（仅改 config.js 数值 / 仅用 ~ 调参器试探，不动结构逻辑）走 **§十** 轻量通道，不重复本硬墙；仅当调参牵动公式/结构/伤害管线时，升级为本节出完整计划。

## 十、数值调参 / 平衡专用流程（激活量化 + 泛化的轻量通道，🔒 反降智）

本流程是 §七 在"纯数值调参"下的专用轻量通道，与 §八 互补，但**绝不降级 AI**——AI 是主动的量化与平衡伙伴，不是抄写员。

### 0. 触发判定
- 仅改 config 数值 / 仅用 ~ 调参器试探 → 走本流程（不触发 §八）。
- 数值**结构变化**或牵动公式/逻辑（改 `Core.Formula`、加字段、改伤害管线）→ 升级 §八 出完整计划。
- 判不准 → 宁按 §八（更严）处理。

### 1. AI 必须主动量化（🔒 禁止降智）
- 用户给"手感/意图"（如"敌人太肉""前期太难"）时，AI **必须推导**具体候选数值，而非等用户报数。推导可追溯：基准值 → 目标 delta → 公式依据。
- 每提案至少 **≥2 候选**，各附权衡（手感曲线/难度坡度/对下游影响）。
- 用 `Core.Formula` / Bus 链路做**波及分析**：改值影响哪些系统、是否被 §六 伤害一致性覆盖。
- §9 未量化值：按 §七 `🟡 // TODO` + 2 候选占位，标债务。

### 2. 反降智护栏（显式禁令）
- ❌ 禁止"用户说 a 改成 b 就直接抄写"：即便接受用户数值，也须附**推导/校验**，确认不破坏 §9 一致性与 §六 铁律。
- ❌ 禁止跳过量化：不允许只回"已调整"而不给具体数字 + 可复核理由。
- ❌ 禁止盲从也禁止静默否决：AI 若认为数值不合理，**必须提更优候选并说明理由**（激活泛化），明示分歧让我选，不替我猜也不闷头照做。
- ✅ 鼓励：基于运行时数据/手感做沙盘推演，主动指出"改 X 顺带让 Y 失衡"的连锁效应——这正是泛化用武之地。

### 3. 轻量提案模板（《调参提案》=）
1. 目标一句话（手感意图，非裸数字）2. 候选数值与推导（≥2，含基准/delta/公式）3. 波及分析（经 Bus/Formula 受影响系统）4. §9 回写项 5. 验收（正常 + ≥1 边界/反向）

### 4. 落地顺序（呼应 §七，不可反）
~ 调参器试探 → 产《调参提案》确认 → 先回写 §9 → 同步 config → 登记 §9 Changelog → 给测试清单。

### 5. 多窗口用法
可新开"平衡窗口"专做本流程（上下文干净防数字淹没），但**共用同一套 §十 规则不得放宽**；需升级 §八 时照常出完整计划。
