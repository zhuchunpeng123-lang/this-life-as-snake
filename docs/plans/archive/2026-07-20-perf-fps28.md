# 计划 · 消除 FPS 28 卡顿（2026-07-20）

## 1. 目标一句话
定位并消除「FPS 28 但帧 1.1ms」的卡顿矛盾——让满屏（敌 50 / 粒子满 / 文字满）稳定 ≥55 FPS。

## 2. 涉及文件及理由
- `14_main.js`（L7 入口 fixed-step 主循环）：加**整帧 CPU 计时**，把 `step()+draw()+ui.update()` 的真实 JS 耗时暴露给 HUD，一锤定音区分「GPU 合成瓶颈」vs「主线程 JS/DOM 瓶颈」（现有 `_frameMs` 只测 draw 命令下发，是盲区）。
- `11_render.js`（L6 渲染）：
  - HUD 加「CPU帧 Nms」读数（与现有「帧 Nms = draw 命令」并列）；
  - `MAX_BACK_W` 2200 → 1600，并暴露为 GM 标量滑条 `RENDER.maxBackW`（1000–2400），让用户自己权衡清晰度/帧率（纯渲染表现，零 gameplay）；
  - `drawEnemies` 主循环加 `inView` 剔除（世界外敌人不画 circle / 标记），降 overdraw 基数。
- `12_ui.js`（L5 UI）：`refreshHUD()` 当前**每帧 `innerHTML` 重建整个 HUD**（含 emoji + 字符串拼接），触发 DOM 重排/重绘且未计入帧时间；改为 **~10Hz 节流**（每 100ms 或每 6 帧一次），零 gameplay 风险。

优先单文件分别改；三文件互不耦合（main 只多传一个读数，render 只读/写自有变量，ui 只改节流），无需跨层直调。

## 3. 具体改动点（分条）
1. `14_main.js` `frame()`：用 `performance.now()` 包住 `while(step)` + `render.draw()` + `ui.update()`，把差值存入 `global.__cpuFrameMs`（或经 Registry/Bus 曝光给 render HUD）。不动循环结构与 STEP。
2. `11_render.js` `drawDebugHud()`：HUD 首行追加 `CPU帧 X.Xms`（取自 main 读数）；原「帧 Nms」保留作 GPU 命令下发对比。
3. `11_render.js` `resize()`：`var MAX_BACK_W = 1600`（原 2200）；同文件顶部 `TUNING`-无关，改在 GM 标量表加 `RENDER.maxBackW` 条目（def 1600，rng 同 viewZoom 风格，dec 0），`resize()` 读 `RT('RENDER.maxBackW', 1600)` 封顶。
4. `11_render.js` `drawEnemies()`：在 `for` 循环开头加 `if (!inView(e.x, e.y, e.radius)) { continue }`（flashT 白闪/本体 circle 一并跳过屏幕外，血条已有 inView 剔除）。
5. `12_ui.js` `refreshHUD()` 调用处（`update()` line 316）：加 `_lastHud` 时间戳，`now - _lastHud >= 100` 才执行 `refreshHUD()`；`heartBreakUntil` 扣心闪烁的即时性由 `offerChoice` 等即时事件仍走原路径，节流只影响分数/时间等慢变信息，体感无差。

## 4. 不动的底层文件
- `02_config.js`：仅经 GM `RT()` 桥读 `RENDER.maxBackW`，不改数值结构与 balance 真理源；
- `03_core.js` / `04_collision.js`：默认不动（本次为渲染/UI/诊断层，未牵动引擎）；
- 伤害管线 / `Core.Formula`：不碰（§六 铁律）。

## 5. 验收标准
- **正常场景**：满屏（敌 50、粒子 213、文字 45）FPS ≥ 55，HUD「画布」回落到 ~1600×900 区间。
- **诊断读数**：掉帧时看「CPU帧」：
  - 若 CPU帧 < 5ms 而 FPS 仍低 → 确证 GPU 合成瓶颈，已靠 maxBackW 滑条进一步下探（1280）可继续回升；
  - 若 CPU帧 > 16ms → 主线程瓶颈（step/DOM），据此再出第二轮计划（可能涉及 collision 遍历，会先告知确认）。
- **边界/反向**：把 `RENDER.maxBackW` 拉回 2400，确认画面恢复清晰但 FPS 下降（证明因果）；拉到 1000 确认帧率回升但更糊（用户自定权衡）。
- 回归：GM「相机缩放」滑条初值 0.80 / 两位小数显示仍正常（上一轮已修，本计划不动 editor）。
- `git diff --stat` 仅含 `14_main.js` / `11_render.js` / `12_ui.js`（core/collision/config 未动）。

## 备注（坦白）
上一轮封顶到 2200 仍卡，说明 2200 对当前设备仍偏重，或瓶颈在 DOM 每帧 innerHTML。本计划两条低风险修复（HUD 节流 + maxBackW 下调并暴露滑条）同时覆盖两种可能，且用「CPU帧」探针给出确定性结论，避免盲修。

## 第二轮（2026-07-20，用户实测数据驱动）

### 实测数据回顾（用户发群）
- FPS 33 / CPU帧 1.5ms / 帧 1.5ms / 画布 1600×900 / 敌 28
- FPS 29 / CPU帧 2.0ms / 帧 1.1ms / 画布 1600×900 / 敌 50
- FPS 45 / CPU帧 3.2ms / 帧 1.5ms / 画布 1600×900 / 敌 37

**结论**：CPU帧全程 ≤3.2ms → 确证瓶颈在 GPU 合成/填充率，非主线程逻辑。第一轮(maxBackW 2200→1600)把 28 拉到 29–45，但仍有偶发尖峰。用户补充两个触发点：①远处假人堆回视有「加载式卡顿」；②浏览器最小化再点开卡顿。

### 根因
1. **火墙常驻 overdraw**：`drawSkillAura` 对**每个蛇身节点**独立 `arc+stroke+fill` 大圆（50 节=50 独立填充 pass 叠加），跟蛇头永远在视口内 → 稳态 GPU 拖帧主因。
2. **假人回视 pop-in**：50 个假人同色，回视时 50 个独立 `fill` 同帧涌入 → overdraw 尖峰（满血不画血条，故「伤害开始那瞬」血条+飘字齐现才是尖峰）。
3. **最小化再点开**：标签页隐藏时 rAF 停、GPU 上下文重置，恢复瞬间 `now-last` 巨大被钳到 0.25 → 主循环 `while` 连跑 ~15 个 `step()` 追帧爆发。
4. 粒子系统已有上限护栏（maxTexts 48 / maxParticles 350 / 帧预算 120，伤害飘字 `low` 优先会被丢弃）→ 飘字风暴非元凶。

### 本轮改动（均在 render / 主循环层，不动 core/collision/§9/伤害管线）
- `11_render.js` 火墙：单 path 连续火管 + 2 次 `stroke`（软火体 + 热边）替代每节点 `arc+stroke+fill` → 50 填充 pass → 2 stroke，消除常驻 overdraw 主因，几何/伤害判定不变。
- `11_render.js` `drawEnemies`：普通本体按色批量 `fill`（50 同色假人→1 次 fill）；白闪/telegraph 特殊件仍单独画；标记/血条第二遍保留原逻辑。
- `14_main.js` `frame()`：大间隔(>0.1s，卡顿/恢复/切后台)直接丢弃追帧（原 clamp 0.25 仍爆 ~15 步）；标签页 `hidden` 时跳过 step+draw 且保持 `last` 新鲜 → 恢复瞬间不爆发。

### 验收标准（用户浏览器手测）
- **正常场景**：满屏稳定 FPS 应明显回升（目标 ≥50，原 29–45）；CPU帧仍 ≤5ms 证明非主线程回归。
- **假人回视**：放一堆假人→跑远→回视，应无「加载式」单帧卡顿（批量 fill 削平尖峰）。
- **最小化再点开**：最小化→点开，应无追帧爆发卡顿。
- **边界**：若仍偶发卡顿，GM「画布上限W」滑条下探 1280/1000 即可再补余量（纯渲染表现）。
- 回归：`RENDER.maxBackW` / `viewZoom` 滑条、T1–T4 开关、HUD 读数均正常。
