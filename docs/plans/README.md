# docs/plans/

按 AGENTS.md §四 / §九，复杂改动在此存放《计划》md（`<需求名>.md`）。
审查窗口 B 按路径读取，做静态核对，输出差异报告（只报告、不改代码）。

## 总索引
所有计划文件的状态、处置、互相 supersede 关系见 **[STATUS.md](./STATUS.md)**。动任何计划文件前先更新它。

## 状态图例
- ✅ **archive/ 已落地**：代码/配置已 commit，特性可用 → 落地后即移入并标记 ✅
- 🗑 **deprecated/ 废弃/被推翻**：判断有误或被新方案取代，**保留不删**（git 可恢复 + 留作警示），移入并注明「为何废弃」
- 🔍 **diagnosis/ 诊断已结**：仅排查未改码，结论已吸收 → 移入
- 🟡 **根目录 active**：待办 / 被阻塞，留根目录持续跟进，直到落地或废弃

## 归档约定（防文件膨胀）
1. **新增计划**：先在本目录建 `<需求名>.md`，并**立即在 STATUS.md 登记一行**（状态=🟡 active）。
2. **落地即归档**：代码 commit 后，把 md 移到 `archive/` 并在 STATUS.md 把状态改为 ✅；落地前置未满足（如 §9 回写未做）则留 🟡 或转 🗑。
3. **被推翻/过时**：移到 `deprecated/` 并写明「被 X supersede / 为何废弃」，不要默默删除（git 历史可追，但文件留底防重蹈）。
4. **纯诊断**：仅排查不改码，结论吸收后移 `diagnosis/`。
5. **移动会断开相对引用**：移动前在 STATUS.md 的「互相关系」列补 supersedes / superseded-by 与原始路径，保证跨文件可追溯。
6. 每个计划/测试清单尽量**自带验收标准与复现步骤**，便于浏览器手测与窗口 B 静态核对。
