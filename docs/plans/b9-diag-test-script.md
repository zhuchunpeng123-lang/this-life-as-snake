## 2. 🔬 本轮测量（决定修法 · 4 项决定性数据）

> 不需要再跑 T1/T3/steamCap 对照（上轮已证它们拦不住掉帧）。本轮**专门制造你说的「2 冰重叠 + 震屏爆炸」最重窗口**，录 4 类数据。

### 测量 ① · 火焰图 script vs render + 最热函数（勾 Memory 看 GC 竖条）
- **操作**：Chrome DevTools → Performance → **勾选 Memory** → 录制 → 在「2 冰重叠 + 爆炸」最重窗口保持 5s → 停止。
- **看**：
  - 是 `script`（逻辑）还是 `render`（draw）占主导？
  - 最贵的**单个函数**是谁（预期候选：① `drawWorld` 里的 `beam`/`blast`/`dart` 循环（render）；② `tickCombos` 蒸汽 AOE 遍历（script）；③ `drawOverlay` 闪核 fill（render））？
  - GC 竖条是否密集（弹性池抖动）？
- **记录**：主导方 `____`；最热函数 `____`；GC 竖条 `密集/稀疏`。

### 测量 ② · 6 数组峰值（行4，看哪个真飙到上千）
- **操作**：在「2 冰重叠 + 震屏爆炸」窗口盯 HUD 行4，录各自**峰值**（尤其爆炸最猛那 1~2 秒）。
- **看**：`beam/blast/dart/flash` 哪个飙到几百/上千？`p/t` 是否早已封顶（350/48）说明不是它们。
- **记录**（峰值）：`p ____  t ____  beam ____  blast ____  dart ____  flash ____`。
- **判读**：若某数组飙到上千 + 测量① render 主导 → 主因是「该数组无活跃数上限」（我的假设）；据此定 cap 值用实测峰值，不拍脑袋。

### 测量 ③ · 蒸汽 AOE 邻居比较峰值（行2 的 `AOE比较`）
- **操作**：同窗口盯 HUD 行2 `引爆(真)` 与 `AOE比较` 峰值。
- **看**：`AOE比较` 是否暴涨（如数千）？`引爆(真)` 是否同步高？
- **记录**：`引爆(真) 峰值 ____`；`AOE比较 峰值 ____`。
- **判读**：若 `AOE比较` 暴涨 + 测量① script 主导 → 主因是蒸汽 AOE 遍历（你的假设）；此时即便给 beams 加 cap 也救不回，需把 `tickCombos` 遍历改空间网格/合并查询。注意：代码已确认 AOE 走 `collision.queryCircle`（空间哈希），但每帧每触发敌都查一次，比较次数 = Σ(触发敌×半径内敌)，量有多大靠本测量定。

### 测量 ④ · T2 单独关屏震是否救场（上轮没测）
- **前置**：回到锁死场景全开状态，记基准 FPS。
- **操作**：`性能诊断 → T2 关屏震` 点成关（行3 T2 变"关"），其余全开，制造「2 冰重叠 + 爆炸」最重窗口。
- **看**：FPS 是否回升？屏震关掉后若仍卡 → 屏震不是主因（它只是 canvas translate，本就便宜）；若回升 → 屏震那一帧在画额外东西，需另查。
- **记录**：T2 关后 FPS `____`、帧ms `____`、是否仍卡 `是/否`。

### 决策矩阵（数据回来对号入座）
| 现象组合 | 修法 |
|---|---|
| 测量②某数组飙到上千 + 测量① render 主导 | 给该数组加活跃数上限（cap 值用实测峰值定） |
| 测量③ AOE比较暴涨 + 测量① script 主导 | `tickCombos` 蒸汽 AOE 遍历改空间网格/合并查询降 O(n²) |
| 两者并存 | 都修 |

- **安全确认（读码）**：`beams/blasts/darts/flashCores` 均为纯视觉——伤害/碰撞由 `hurt/hurtCombo/applySlow` + `04_collision` 即时结算，这四个数组不承载任何 gameplay 状态，未来落 cap 时 `arr.shift()` 淘汰最旧不会掉伤害。
- **综合排查结论**：`05_particle.js` 无「伤害飘字排队」机制（搜 queue/pending/buffer=0）；飘字即时 `spawnText` 且受 `maxTexts=48` 上限约束；真正的无界增长风险就是那 4 个无活跃数上限的 VFX 数组。
